# Ford 项目 Jira 问题分析器

该技能专门用于处理和分析 Mega 公司 Ford 项目的 Jira 问题单。请严格遵循以下标准工作流进行处理。

## 严格执行顺序与日志驱动分析（Strict Sequence & Log-Driven Analysis - 🚨 核心准则）
- **🚨 禁止提前分析**：不要自作主张，在没有完成日志下载、解压、合并、PID 提取等所有前置步骤前，**严禁**调用子 Agent（如 `explore`, `oracle` 等）根据 Bug 描述盲目开始代码分析。这会导致定位不准并浪费资源。
- **日志先行**：所有的代码逻辑分析和子 Agent 调用，**必须**建立在已经提取出的“进程关键日志”和“系统合并日志”的基础之上。必须结合日志中的报错堆栈、关键时间点的执行流，再加上 Bug 描述，再去检索和分析源码。这样能更精准地定位出问题的代码位置。
- **必须串行执行**：工作流中的步骤 [获取日志] -> [模块/PID识别] -> [提取进程日志] -> [代码/逻辑分析] **必须严格串行**。前一步未完成（未生成对应的日志文件），禁止启动后一步。
## 核心工作流：问题分类与分流

当你接收到一个 Jira 单号时，**第一步必须是判断问题的类型并决定走哪条工作流**。

### 1. 提取基础信息与粗略分类
- **识别车型**：使用 `scripts/parse_jira.py` 脚本提取单号信息并判断车型：
  - **Cardiff**: 8255平台，前缀 `CF-`
  - **Monterey**: 8155平台，前缀 `MRY-`
  - **PebbleBeach**: 8155平台，前缀 `PB-`
- **获取详情**：使用 `jira_get_issue` 获取标题、描述和 Issue Type。
- **初步分流**：
  - **如果是 Task**：向用户提示“此单是一个任务单(Task)，不需要立即处理。”，并询问用户是否继续。
  - **如果是 Bug**：向用户提示“此单是一个 Bug 单，需要立即处理。”，并进入下一步的深度分类。同时在 `<jira单号>.md` 中记录基础信息。
  - **🚨 Reopen 状态特殊处理**：检查 Bug 状态是否为 **Reopen**。
    - **特征**：Bug 状态显示为 Reopen，通常意味着之前的修复在测试验证中失败。
    - **获取新信息**：**必须读取最新的 Comment**。测试人员通常会在最新的 Comment 中描述新的问题表现、提供新的发生时间点并上传新的日志附件。
    - **强制要求**：在分析 Reopen 问题时，**严禁**只参考原始 Description。**必须以最新 Comment 中的时间点和描述为准**，并下载**与 Comment 发布日期匹配**的新日志附件进行分析。

### 2. 客户 Bug 识别与特殊处理（如果适用）
- **识别规则**：检查 Jira 标题是否包含 `【客户】`、`【客户问题】` 或 `【客户Bug】` 等关键字。
- **客户 Bug 专属工作流**：如果确认为客户 Bug，必须注意以下几点：
  1. **描述模糊**：客户上报的问题描述可能不专业或比较模糊，不能仅仅依赖字面意思，必须结合上下文和日志来判断真实情况。如果实在无法理解，需主动向用户澄清。
  2. **日志获取方式变更**：客户 Bug 的日志通常是通过网盘分享，而不是直接作为附件上传在 Jira 中。
  3. **日志处理动作**：
     - **优先检查附件**：使用 `jira_list_attachments` 检查当前 Jira 单是否包含日志附件。
     - **附件下载**：如果有日志附件（可能是多个以 `.gz` 结尾的文件，或者 `.zip`, `.rar` 等打包文件），必须将它们**全部下载**到 `~/Downloads/<jira单号>` 目录下，并进行解压处理。
     - **询问路径（后备方案）**：如果在 Jira 附件中**没有发现任何日志文件**，必须主动询问用户：“这是一个客户 Bug，我在附件中没有找到日志，请问您从网盘下载的日志文件本地存储路径是什么？”
     - **检查、拷贝与清理**：如果从用户处获得了路径，检查该路径是否有效，拷贝到 `~/Downloads/<jira单号>` 目录下，然后**必须删除原始位置的日志文件**。
### 3. Bug 深度分类与工作流选择
通过阅读 Jira 标题和描述（并结合前置上下文），必须将 Bug 归入以下两类之一：

#### 分类 A：纯 UI 效果调整类（短路工作流）
- **特征**：标题和描述涉及“显示不清晰”、“无法淡入/淡出”、“颜色不对”、“边距问题”、“截断”、“圆角错误”等纯视觉或动画样式问题。
- **执行流程**：
  1. **不下载日志**：这类问题不需要日志证明，不要执行耗时的日志下载与提取操作。
  2. **静态搜索**：根据描述，通过代码或者布局 xml 文件的静态搜索，找出**可能需要修改**的 UI 元素、控件或布局属性。
  3. **提示用户并短路**：向用户汇报：“这是一个纯 UI 效果问题，建议修改的文件路径和可能的位置是 xxx。请自行确认设计稿并修改代码。”
  4. **交出主动权**：终止后续的自动化修改与提交流程，等待用户自己完成修改。

#### 分类 B：逻辑 Bug 类（完整工作流）
- **特征**：除了分类 A 以外的所有问题（如崩溃、功能未生效、无响应、状态错误等）。
- **执行流程**：执行下面的**【标准逻辑 Bug 处理流程】**。
  - *如果无法明确区分是哪类问题，必须向用户询问确认后再决定走哪条工作流。*

## 标准逻辑 Bug 处理流程

### 1. 明确时间点（⚠️ 强制检查点）
- 必须明确问题发生的具体时间点（精确到分钟即可，如：15:05）。
- **🚨 强制性规则**：读取 Jira 描述后：
  - **如果有明确时间点**：记录该时间点，继续后续步骤。
  - **如果没有明确时间点**：**必须立即停止当前所有操作，向用户询问**："Jira 描述中未找到问题发生的具体时间点，请提供该 Bug 发生的准确时间（精确到分钟），以便提取对应时间段的日志进行分析。"
  - **🚫 禁止行为**：在没有获得用户确认的时间点之前，禁止擅自猜测时间、禁止下载日志、禁止进行任何日志分析操作。

### 2. 日志获取与提取
- **获取日志**：统一存储在 `~/Downloads/<jira单号>` 目录下。
  - 内部单：使用 Jira 工具（`jira_list_attachments`, `jira_download_attachment`）下载附件，**并确保所有附件都已下载完成**，完成后，保存到 `~/Downloads/<jira单号>` 目录下，然后清理掉jira工具下载的临时文件。
  - **🚨 Reopen 单日志匹配**：如果是 Reopen 状态，**必须**对比附件列表与最新 Comment 的日期，仅下载并合并与最新 Comment 时间戳最接近的日志文件。避免使用旧日志导致分析结论错误。
  - 客户单：优先检查并下载附件日志（注意可能有多个）；若无附件，则按“客户 Bug 专属工作流”询问用户路径并拷贝处理。
- **提取日志**：在 `~/Downloads/<jira单号>` 目录下解压（支持 `.zip`, `.gz`, `.rar` 格式），并在 `android` 目录寻找 `.gz` 或 `logcat.log`。注意 `logcat.log` 通常是最后一段日志，对于附件中直接提供的多个 `.gz` 日志文件，要确保它们都在分析范围内。
- **合并过滤**：根据发生时间点，向前后各取 4 分钟（共 8 分钟）的日志。使用 `scripts/extract_logs.py` 将这些文件**合并为一个完整的系统日志文件**（`merged_<jira单号>_<时间戳>.log`）。**完成后立即在 `<jira单号>.md` 中更新日志合并状态。**

### 3. 模块归属与进程识别
- **明确模块与范围限制（🚨 核心限制）**：首先根据 Jira 描述中的关键词判断问题模块。**本技能目前支持 CardiffSystemUI, CardiffUser 和 CardiffEngineeringMode 模块的分析。**
  - **在支持范围内**：
    - "下拉快捷面板"、"快捷控制"、导航栏"、"状态栏"、"HOME键" -> `/home/rentianxin/Code/JMC/cardiff/CardiffSystemUI` (包名: `com.android.systemui`)
    - "个人中心" -> `/home/rentianxin/Code/JMC/cardiff/CardiffUser/` (包名: `com.mega.user`)
    - "工程模式" -> `/home/rentianxin/Code/JMC/cardiff/CardiffEngineeringMode/` (包名: `com.mega.engineeringmode`)
  - **在支持范围外**：如果判定为其他非上述模块，**必须立即告知用户**：“检测到该问题不在本技能的支持范围内。请确认是否需要我进行尝试性分析，或由您自行处理。”，并停止标准自动化流程。
- **阅读 AGENTS.md**：明确模块并在终端切换到对应模块的根目录后，在开始阅读和修改源码之前，**必须首先读取并分析该模块根目录下的 `AGENTS.md` 文件**。该文件包含了模块的功能所属、关键编译指令和重要上下文，可以防止执行无效的编译命令或触发错误操作。
- **提取进程号(PID)**：在合并日志中找出对应模块的包名对应的 PID（如 `com.android.systemui (pid 2778)` 或 `2778/com.android.systemui`）。如果找不到，主动询问用户。
- **保存进程日志**：提取该 PID 的专属日志存为 `process_<模块名>_<PID>_<时间戳>.log`，保存到 `~/Downloads/<jira单号>/` 目录下。**记录 PID 信息到 `<jira单号>.md`。**

### 4. 日志分析与定责（🚨 核心避坑）
- **进程级分析**：看 `process_*.log` 分析模块内部逻辑。
- **系统级跨进程分析**：涉及组件启动、输入分发等行为，**必须看 `merged_*.log`** 原始系统日志，否则会漏掉 system_server、其他应用行为等关键日志。
  - 如果发现目标应用无响应、系统整体卡死、或其他非本技能支持模块（非 SystemUI/User/EngineeringMode）的问题，**必须遵循以下极度严格的定责流程**：
    1. **罗列无责证据（🚨 核心要求）**：必须尽可能多地罗列证据，**证明本模块执行逻辑是正常的**。不能仅给出结论，必须能够**说服**其他模块的开发人员。证据必须包含：
       - **日志原文**：直接贴出关键的日志行原文（带时间戳）。
       - **逻辑解释**：解释该日志行代表本模块已完成的任务（例如：Intent 已发出、Callback 已执行、数据已正确透传）。
       - **对比说明**：指出对方模块在日志中对应的缺失响应或异常行为（例如：对应的 Activity 未接收到 Intent、目标服务未返回结果）。
    2. **整理分析报告**：在 `<jira单号>.md` 中汇总上述证据，形成一份详尽的“外部定责报告”。
    5. **严禁分析非负责模块方案**：**严禁**对判定为非我方负责（非 CardiffSystemUI/User/EngineeringMode）的问题进行任何修复方案的分析、代码搜索或修改尝试。
    6. **告知用户并同步 Jira**：向用户明确汇报：“经过深入分析，此问题证明不在我方负责的模块范围内（仅负责 CardiffSystemUI/User/EngineeringMode）。定责理由如下：[详细理由及日志原文证据]。建议您可以将此问题转交给相关模块同事处理。”
    7. **询问 Jira 同步**：**必须询问用户**：“是否需要我将这份详尽的定责证据以 Comment 形式同步到 Jira 单中，以便您将问题转交给相关模块同事处理？”（此时同步的格式不限，以讲清楚问题和理由为准）。
    8. **记录结论**：将最终结论及证据同步更新到 `~/Downloads/<jira单号>/<jira单号>.md` 中。
### 5. 分析结论汇报与方案确认（🚨 核心强制阻断点）
## 严格的阶段性用户交互流（Strict User Interaction Flow - 🚨 强制准则）

为了确保操作的准确性和用户的掌控权，以下三个步骤**严禁合并**，每一阶段必须进行独立的确认交互。**禁止通过一个问题请求用户同时确认多个步骤。**

### 第一阶段：分析结论与代码修改确认
- **动作**：完成日志分析，提出修改方案。
- **交互**：向用户展示分析报告（含日志证据和拟修改代码），并询问：“以上是我的分析结论和修改方案，请问是否确认？确认后我将开始自动修改代码并执行编译验证。”
- **阻断**：**必须**收到用户对“修改代码”的明确授权后，方可执行。**禁止**在用户未回复前执行任何写操作或编译指令。

### 第二阶段：Git Commit 确认
- **动作**：代码修改完成且**编译通过**。
- **交互**：必须在编译成功后，调用 `git-commit-message` 生成提交信息并展示给用户，询问：“代码已修改并编译通过，是否确认以上述信息执行 Git 提交？”
- **阻断**：**必须**收到用户对“执行 Commit”的独立授权后，方可执行 `git commit`。**严禁**在第一阶段确认时顺带执行此步骤。

### 第三阶段：Jira Comment 确认
- **动作**：Git 提交成功。
- **交互**：向用户展示将要追加到 Jira 的 Comment 内容，询问：“Git 提交已成功，是否需要我将修复信息以 Comment 形式同步到 Jira 的 Comment 中？”
- **阻断**：**必须**收到用户对“回填 Jira”的独立授权后，方可执行 `jira_add_comment`。
- **🚨 Comment 格式要求**：提交到 Jira 的 Comment 必须**严格**遵循以下模板（注意：“合入时间和分支信息”部分只保留标题，不要填写任何内容）：
  ```text
  问题原因：（这里填写具体的问题原因分析）
  修改方案：（这里填写具体的代码修改方案）
  合入时间和分支信息：
  ```

**🚨 核心禁令**：严禁“自作聪明”地跨越阶段。如果用户只同意了第一步，绝对不能自动执行第二、三步。每个步骤都是一个独立的决策点。禁止“一问三答”。

## 项目通用知识 (Project General Knowledge)

### 知识 1：可见可说 (Visible Speech) 功能
- **功能定义**：Android 车机系统中，一些 UI 元素（如按钮）配置了 `ContentDescription` 属性。当系统 `accessibility`（辅助功能）开启时，用户可以通过语音指令直接触发这些元素的点击行为。
- **触发方式**：例如，输入语音指令“隐私政策”或“点击隐私政策”，系统会查找 `ContentDescription` 匹配的元素并执行 `performClick()`。
- **核心原则**：原则上，当按钮被更高层级的界面（如弹窗、遮罩、其他 Activity）完全遮挡时，是不应该通过可见可说功能触发点击的。如果发生了遮挡点击，通常说明层级管理或可见性判断存在 Bug。
一旦确认问题是可见可说类的问题，需要参考文档 可见即可说AI诊断指导.md 进行分析和定位问题。
